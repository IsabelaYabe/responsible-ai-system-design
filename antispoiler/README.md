# antispoiler

Position-bounded retrieval for a spoiler-free reading assistant, with an
evaluation harness (bounded vs. unbounded baseline, LLM-as-judge, human-anchored
agreement).

## Idea

A spoiler is a function of *position*, not just content. The mechanism enforces
the reader's knowledge boundary at the **retrieval layer**: chunks are tagged
with `chapter_index`, and a hard pre-retrieval filter drops anything past the
reader's position. The model physically cannot see — and so cannot reveal —
future text, regardless of the prompt. See the report for the full rationale.

## Layout

| Module | Responsibility |
|---|---|
| `config.py` | Constants + credential resolution (Colab `userdata` → `.env`/env) |
| `llm_client.py` | Model-agnostic client; separate answerer (Haiku) / judge (Sonnet) |
| `book.py` | Fetch + chunk a book by chapter (`Chunk`, `chunk_text`) |
| `index.py` | FAISS cosine embedding index over chunks |
| `retrieval.py` | Smart retrieval (entity-lexical + embedding) + `recall_retrieve`; `reader_pos=None` ⇒ unbounded |
| `qa.py` | Reading-companion QA prompt + `ask()` |
| `respond.py` | Product dispatcher: `(selection, intention, position)` → reply; `respond_detailed` for the eval |
| `qgen.py` | LLM-generated items — tiered QA questions **and** per-intention items |
| `judge.py` | LLM-as-judge: safety `VERDICTS` **and** quality `QUALITY_LABELS`, each a single source of truth |
| `evaluate.py` | Two eval loops (generic QA + per-intention), metrics, Cohen's κ for both axes |

The notebook `notebooks/antispoiler_eval.ipynb` is a thin orchestration layer;
it is generated by `scripts/build_notebook.py` — edit the package, not the JSON.
It has two parts: **Part A** evaluates the generic companion QA prompt; **Part B**
evaluates the product's three retrieval-bearing intentions (define, contextualize,
recall) per-path, scoring quality on all three and spoiler-safety (bounded vs.
unbounded) on the two position-sensitive ones. Paraphrase is out of scope.

## Setup (Apple Silicon / arm64)

The repo's miniconda base is x86 (Rosetta), where torch is capped at 2.2.2. Use a
**native arm64** env:

```bash
CONDA_SUBDIR=osx-arm64 conda create -y -n antispoiler-arm python=3.11
conda run -n antispoiler-arm conda config --env --set subdir osx-arm64
conda run -n antispoiler-arm pip install -r requirements.txt
```

Put credentials in a `.env` at the repo root (git-ignored):

```
API_KEY=sk-ant-...
HF_TOKEN=hf_...
```

## Run

```bash
# unit tests (no API, no model download)
conda run -n antispoiler-arm python tests/test_logic.py

# full eval in the notebook
conda run -n antispoiler-arm jupyter notebook notebooks/antispoiler_eval.ipynb
```

## Demo (interactive web app)

The notebook is the *evaluation* harness; `app.py` (repo root) is the
*demonstration*. It models the real product interaction — the reader highlights
a span, picks an intention (Define / Paraphrase / Contextualize / Recall), and
gets a position-bounded reply. The book renders only up to the reader-position
slider, so you can only select text you've "read"; moving the slider makes
spoilers appear/vanish.

```bash
# from the repo root; needs the .env API key, same as the notebook
conda run -n antispoiler-arm uvicorn app:app --reload --port 8000
# then open http://127.0.0.1:8000
```

Notes:
- **First request is slow:** at startup it downloads the embedding model and
  indexes the whole book once.
- One extra dependency beyond the eval: `fastapi` + `uvicorn` (in
  `requirements.txt`). The frontend pulls `marked` from a CDN to render markdown,
  with a plain-text fallback if offline.
- Layout: `app.py` (FastAPI routes) + `static/index.html` (single page) +
  `antispoiler/respond.py` (the `(selected_text, intention, reader_position)`
  dispatcher). Same backend as the eval — `book` / `index` / `retrieval` / `llm_client`.

## Validation notes

- **De-confounded baseline:** bounded and unbounded share the same retrieval
  strategy; only the position filter differs, so any leakage difference is
  attributable to the filter.
- **Judge bias:** the judge uses a stronger, different model than the answerer
  (self-preference bias), with few-shot anchors. Trust the leakage numbers only
  after checking `judge_human_agreement` (Cohen's κ vs. human labels).
- **Scope:** single book, single reader position, small question set — a proof of
  concept, not a statistical claim.
